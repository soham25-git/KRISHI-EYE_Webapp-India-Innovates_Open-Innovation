## 🚀 How to Deploy for Public Use
To move KRISHI-EYE from `localhost` to public URLs accessible globally, you must decouple the Next.js frontend from the NestJS backend and host them on dedicated cloud infrastructure.

### 1. Deploy the Backend Database
- You are currently falling back to a local SQLite file (`dev.db`). For production, provision a managed PostgreSQL database.
- **Providers:** Supabase, Neon.tech, or Render PostgreSQL.
- **Action:** Update the `DATABASE_URL` environment variable in your production environment to point to this new remote cluster.

### 2. Deploy the APIs (`apps/api` & `apps/ai-service`)
- NestJS and FastAPI require standard Node.js/Python server environments (not purely serverless).
- **Providers:** Render, Railway.app, or DigitalOcean App Platform.
- **Action:**
  - Create a new service on **Railway.app** pointing to the `apps/api` folder.
  - Set `NODE_ENV=production` and map the remote `DATABASE_URL`.
  - Repeat this for the `apps/ai-service` utilizing a Python environment.
  - *Crucial:* Ensure your backend CORS settings exclusively white-list your future frontend domain.

### 3. Deploy the Next.js Frontend (`apps/web`)
- The Next.js app should be hosted on a CDN-backed edge network.
- **Provider:** Vercel (highly recommended, native support).
- **Action:**
  - Connect your GitHub repository to Vercel.
  - Set the Root Directory to `apps/web`.
  - Set the `NEXT_PUBLIC_API_URL` environment variable to the public URL generated by your Backend deployment in Step 2.

## 🛠️ Next Steps for True Production Readiness
Before routing real farmers to your public URL, the following technical gates must be passed:

1. **Dashboard 404 Hotfix:**
   - **Issue:** The auth flow routes users to the root `/` or `/dashboard`, both of which are currently 404ing (despite the `(dashboard)` route group existing).
   - **Action:** We need to explicitly expose an `index.tsx` or `page.tsx` mapping directly to the dashboard components so the primary landing page renders upon login.

2. **Security & Session Hardening:**
   - **Issue:** The demo relies on an OTP bypass (`123456`) and local storage for JWT tokens.
   - **Action:** Ensure the Demo Bypass is strictly tied to `NODE_ENV !== 'production'`. Move JWT storage from `localStorage` into **Secure HTTP-Only cookies** to prevent XSS (Cross-Site Scripting) token theft.

3. **Database Migration to PostgreSQL:**
   - **Issue:** Local SQLite will overwrite or corrupt under concurrent multi-tenant loads.
   - **Action:** Execute `prisma migrate deploy` against a real remote PostgreSQL instance.

4. **Public Domain Configuration:**
   - Purchase a domain (e.g., `krishi-eye.com`) and map the DNS records to your Vercel frontend. Ensure SSL certificates are active globally.
